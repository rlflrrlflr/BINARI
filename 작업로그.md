### [11] 아티팩트 판결 복구 (v19) — 내가 버렸던 내장 API 되살림
- **요청**: "이전엔 아티팩트에서 API 물려서 됐다. 다시 그렇게 해줘. 배포 전제하지 말고."
- **판단**: 사용자가 옳다. 아티팩트에는 **window.claude.complete(문자열 in/out)** 내장 API가 있고, 이게 "물려있던" 정체. v16에서 내가 이걸 버리고 /api/judge로 바꾼 게 근본 원인. 앞서 "window.claude.complete는 2콜·캐싱 훼손이라 안 함"이라던 내 판단을 뒤집음 — 초기 단계의 최우선 가치는 '지금 폰에서 되는 것'이고, 캐싱은 그 다음. 판결 경로를 3-way로: complete(있으면 최우선)→server→direct. system+messages를 한 프롬프트로 병합, 콜1/콜2는 프롬프트 텍스트로 구분. **검증을 사용자에게 떠넘기지 않기 위해** window.claude.complete를 mock 주입해 헤드리스에서 콜1·콜2 판결 렌더까지 확인(4/4). (첫 시도에서 캡처 타이밍이 카드 애니메이션보다 일러 오탐 → 대기 로직으로 수정.)
- **결과**: `app/e2e/verdict.mjs` 신설(판결 회귀 4/4), 스모크 21/21 유지. v19 스냅샷 승격. 배포 환경에선 complete 부재→자동으로 /api/judge. **아티팩트·배포 양쪽 다 지원.** 단 아티팩트 런타임에 window.claude.complete가 실제 존재하는지는 사용자 실측으로 최종 확인 필요(mock은 우리 코드가 올바르게 호출·파싱함만 증명).

- **구현**:
  - 반복1(음력) — `lunar2solar`(기존 LUNAR표·solar2lunar의 역) 신설, 음/양력 토글+윤달 체크, doReveal에서 음력이면 양력 정규화 후 모든 계산(사주·달·별자리·수비학·토정·삼재)에 양력 전달. birth에 정규화된 양력 저장(원본 음력은 lunarNote). 구버전 저장분은 cal 부재→양력 기본으로 호환.
  - 반복2(이름) — birth.name 선택 입력(맨 위, "너를 뭐라고 부를까?"), 프로필 상단 '호칭' 주입 + SYS "결정적 순간에 이름 부른다(B ○○아·A ○○님)" 규칙, 리빌 멘션 "그래 — 서연, 너는 원래…". 판결 미반영(친밀도 전용) — 혈액형과 다른 점은 값이 '이름 불러주는 존재'라는 정서.
  - 반복3(성별→대운) — `daeun` 신설: 양남음녀 순행/음남양녀 역행, 대운수=순행 다음 節·역행 이전 節까지 일수/3(태양황경으로 節 크로싱 탐색), 현재 나이로 대운 간지(월주±step) 산출. 성별 선택 시에만 계산, 프로필 타이밍 층에 "대운(현재 인생 시기): 무인(토) 33~42세 역행"으로 주입. SYS는 대운을 별도 축 신설 없이 사주 근거에 녹이도록 지시 → 판결문 형식 보존.
- **검증**: 만세력 24/24 — 음↔양 왕복 6/6, 설날 2건(2000.2.5·1993.1.23), 윤달 구분, 설날생 사주년(음력정월→양력1.23→입춘전→임신년), 대운 방향(양년 남 순행/여 역행)·대운수 범위·pre·첫대운=월주±1. 전체 55/55. 프로필 실측(headless로 window.claude.complete 가로채 프롬프트 원문 확인): 호칭/성별/대운 3항목 정상 주입. e2e 셀렉터는 이름 입력(input.in.wide) 추가로 밀려서 `input.in:not(.wide)`로 수정.
- **주의(기록)**: 대운 간지는 알고리즘(방향·대운수·간지 스텝)을 내부 검증했으나 외부 만세력과의 대조는 사용자 실측 권장. 대운수는 세는나이 근사(10년 버킷엔 무해). 이름은 수호신 시드에 미포함(형상 불변).
