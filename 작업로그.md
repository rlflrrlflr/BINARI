### [11] 아티팩트 판결 복구 (v19) — 내가 버렸던 내장 API 되살림
- **요청**: "이전엔 아티팩트에서 API 물려서 됐다. 다시 그렇게 해줘. 배포 전제하지 말고."
- **판단**: 사용자가 옳다. 아티팩트에는 **window.claude.complete(문자열 in/out)** 내장 API가 있고, 이게 "물려있던" 정체. v16에서 내가 이걸 버리고 /api/judge로 바꾼 게 근본 원인. 앞서 "window.claude.complete는 2콜·캐싱 훼손이라 안 함"이라던 내 판단을 뒤집음 — 초기 단계의 최우선 가치는 '지금 폰에서 되는 것'이고, 캐싱은 그 다음. 판결 경로를 3-way로: complete(있으면 최우선)→server→direct. system+messages를 한 프롬프트로 병합, 콜1/콜2는 프롬프트 텍스트로 구분. **검증을 사용자에게 떠넘기지 않기 위해** window.claude.complete를 mock 주입해 헤드리스에서 콜1·콜2 판결 렌더까지 확인(4/4). (첫 시도에서 캡처 타이밍이 카드 애니메이션보다 일러 오탐 → 대기 로직으로 수정.)
- **결과**: `app/e2e/verdict.mjs` 신설(판결 회귀 4/4), 스모크 21/21 유지. v19 스냅샷 승격. 배포 환경에선 complete 부재→자동으로 /api/judge. **아티팩트·배포 양쪽 다 지원.** 단 아티팩트 런타임에 window.claude.complete가 실제 존재하는지는 사용자 실측으로 최종 확인 필요(mock은 우리 코드가 올바르게 호출·파싱함만 증명).
