### [4] 개발 루프 (v16, 반복 1~8)
- **요청**: 계획대로 개발, loop 엔지니어링.
- **판단**: 코드 사실이 정한 순서 — 0단계(아티팩트 탈출: 배포가능화+데드엔드 수리)를 리텐션(1단계 B1~B7)보다 먼저. 매 반복은 **transform 스크립트로 "정확히 1회 매칭" 검증 후 치환** → 잘못된 위치 수정 원천 차단. 매 반복 `vite build` 통과를 게이트로.
  - 왜 계정 대신 localStorage(B1): 가입 마찰만 얹고 지금 단계 추가가치 0. 멀티기기·결제 필요해지는 2단계 이슈.
  - 왜 Supabase 0단계 배제: 저장할 서버 데이터가 아직 없음. 비개발자에게 두 번째 학습곡선.
  - 왜 속결 모드 역전(B5): 3초짜리 결정에 6단계 동전 의식은 데일리 사용을 죽임. 프롬프트의 "유저가 청한 경우만 주역"과 UI를 일치.
- **결과**: 반복 1(0단계)→2(B1 기억)→3(B2 데일리)→4(B4 부적내보내기)→5(B3 되물음)→6(B5 속결)→7(B6판결록+B7스트릭)→8(런타임검증). 반복 8에서 Playwright 모바일 뷰포트 스모크 **18/18 통과**(온보딩·속결/의식 실패복구·재회·데일리). v16 스냅샷 03 승격, v15는 09로.

- **요청**: 레퍼런스 영상 3개 첨부. 지금 버전 불만족 이유/개선법/중독 안 되는 이유/이펙트가 적용 안 되는 이유/보는 사람 심리.
- **판단**: 영상 프레임 실사 → 3개 모두 **GPU 파티클(수십만~수백만) + 유속장(flow field) + 잔상 누적 + 블룸**. 현재 GuardianCanvas는 Canvas2D 1,150~1,600 입자가 **제자리 사인파 진동** → "살아있음"의 핵심인 흐름·인과가 없음. 노력 문제가 아니라 Canvas2D+단일파일 선택의 물리적 천장.
